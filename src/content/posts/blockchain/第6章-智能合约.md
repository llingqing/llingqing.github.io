## 6.1 智能合约及其背景

### 6.1.1 智能合约的定义

智能合约指能够**自动执行合约条款的计算机程序**，其概念由 **Nick Szabo（尼克·萨博）** 于 **1996 年**提出。具有三大特性：

| 特性 | 说明 |
|------|------|
| **事件驱动** | 以外部输入（如投币）作为触发条件，驱动合约动作执行 |
| **价值转移** | 外部输入价值（如钱币），合约输出商品，完成价值交换或转移 |
| **自动执行** | 履约行为完全自动化，不需要人为干预 |

### 6.1.2 自动售货机类比及其局限性

以自动售货机为例说明智能合约，但其存在中心化导致的弊端：

- **容易被篡改**：可以人为篡改售货机程序，使其免费出售商品
- **出错后难以追溯恢复**：恶意篡改人已掌握整个机器的控制权，源头无法追溯
- **依赖执行环境的可靠性**：一旦断电或机器故障，投币后无法获得商品

> 核心结论：智能合约执行所依赖的"中心化"环境导致了**易被篡改、出错后难以追溯恢复**等弊端。

### 6.1.3 区块链智能合约的优势

将智能合约应用到区块链上，能使其具备更高可靠性。以以太坊为例：

- 多节点组成的 P2P 网络中，节点维护共同的区块链数据
- 通过区块链上的交易进行智能合约的**创建、调用、结束**等操作
- 多节点维护的区块链状态一致，因此多节点上运行的智能合约的过程和结果也一致

| 对比维度 | 中心化智能合约 | 区块链智能合约 |
|---------|-------------|-------------|
| 执行环境 | 单一中心服务器 | 分布式多节点 |
| 可篡改性 | 容易被篡改 | 难以篡改 |
| 验证机制 | 无多方验证 | 多方验证 |
| 追溯性 | 难以追溯 | 可追溯 |
| 可靠性 | 依赖单一环境 | 去中心化高可靠 |

### 6.1.4 区块链智能合约的范围

- 比特币中每笔交易所附带的脚本，某种程度上也可认为是智能合约，但**不是图灵完备的**
- 常见的区块链智能合约一般指以太坊、Hyperledger Fabric 等平台中运行的**图灵完备**的智能合约

## 6.2 Solidity 语言

### 6.2.1 背景

Solidity 是一种用于实现智能合约的**面向对象的高级语言**，语法接近 **JavaScript**。设计目标是为编写以太坊上的智能合约，因此部分语法与以太坊工作原理是高度耦合的。

**与其他智能合约语言对比：**

| 语言 | 平台 | 特点 |
|------|------|------|
| **Solidity** | 以太坊 | 目前较成熟、流行，语法接近 JavaScript |
| Serpent | 以太坊 | 较早出现，类似 Python |
| Vyper | 以太坊 | 更注重安全性，语法类似 Python |
| Mutan | 以太坊 | 较早的语言（已较少使用） |
| Golang | Hyperledger Fabric | 通用语言编写链码 |

**Solidity 特殊之处：**
- 特殊类型：`address`、`event` 等
- 特殊关键字：`payable`、`now` 等
- 需要对变量进行具体存储位置的定义（`storage` / `memory`）

**学习资源：**
- GitHub: https://github.com/ethereum/solidity
- 在线编译器（Remix）: http://remix.ethereum.org/
- 英文文档: https://docs.soliditylang.org/
- 中文文档: https://solidity-cn.readthedocs.io

### 6.2.3 基础语法

#### 1. 常用类型

Solidity 是**静态类型语言**，每个变量都需要被定义成能被编译器识别的特定类型。

| 类型 | 声明方式 | 说明 |
|------|---------|------|
| **布尔类型** | `bool` | 取 `true` 或 `false` |
| **整型** | `int` / `uint` | 有符号/无符号整型，可加位数如 `int8`、`uint256`，位数可取 8, 16, ..., 256，未声明默认取最长位数即 `int256` |
| **地址类型** | `address` | 以太坊账户地址，实际存储为 **20 字节**的值 |
| **定长字节数组** | `bytes1` ~ `bytes32` | 由 `byte` + 数组长度声明 |
| **字符串（变长字节数组）** | `bytes` / `string` | 变长类型 |
| **数组** | `类型[]` | 可通过 `push` 或 `length` 操作 |
| **映射** | `mapping(键类型 => 值类型)` | 键类型不可以是映射、变长数组、合约、枚举及结构体 |
| **结构体** | `struct` | 将多个基本变量组合为一个结构 |

**地址类型（address）的成员变量/函数：**

| 成员 | 作用 |
|------|------|
| `<address>.balance` | 获取该地址的余额（uint256） |
| `<address>.transfer(uint256 amount)` | 向该地址转账，失败则回滚整个交易 |
| `<address>.send(uint256 amount)` | 向该地址转账，失败则返回 false |
| `<address>.call{...}()` | 向该地址发出底层的合约调用消息 |
| `<address>.callcode{...}()` | 底层调用（已不推荐） |
| `<address>.delegatecall{...}()` | 底层委托调用 |

**映射（mapping）底层存储特点：**
- 在以太坊状态树底层存储中，实际上**不存储键的值**，而是存储键的 **keccak256 哈希值**
- 目的是便于减小存储体积的同时查询实际的值
- 与数组不同，映射**没有长度**

**变量声明补充：**
- 可指定 `public` 或 `private`：控制是否可被合约外部调用获取查询
- 可指定存储位置为 **`storage`**（存储在状态树中）或 **`memory`**（存储在内存中）
- 不同存储位置涉及 **Gas 计费**的不同
- 每个变量有默认的存储位置，当编译器无法默认识别时，会提示开发者进行修改

#### 2. 函数类型

函数声明形式：

```
function (参数) {public|private|internal|external} [pure|constant|view|payable] [returns (返回值)]
```

**可见性修饰符：**

| 类型 | 作用 |
|------|------|
| `public` | 对合约内外均可见，即可被外部的人或合约调用 |
| `private` | 仅对合约内可见 |
| `internal` | 仅对内部可见，且对继承了该合约的子合约可见 |
| `external` | 仅对外部可见，仅能通过 `合约.函数` 的形式调用 |

**行为修饰符：**

| 类型 | 作用 |
|------|------|
| `pure` | 不允许函数读取或写入状态 |
| `view` / `constant` | 不允许函数写入状态，但可读取状态 |
| `payable` | 允许函数接收以太币。注意：调用合约时，对不带有 payable 的函数进行带有以太币的转账将导致调用失败 |

#### 3. 特殊函数类型

**(1) 构造函数（constructor）**

- 由 `constructor` 关键字声明的**可选函数**
- 只在**合约部署时运行一次**
- 由于区块链空间资源宝贵，构造函数的字节码**不会被保存在区块链上**
- 典型用法：将部署合约的地址（`msg.sender`）赋值给 `owner` 变量，实现权限控制
- 早期 Solidity 版本中构造函数是与合约同名的函数，因容易出现笔误导致安全问题（如大小写错误使任何人都可修改 owner），该写法已逐渐被抛弃

**(2) 回退函数（fallback）**

- 一个智能合约**最多只能有一个**回退函数
- 声明方式：`fallback() external [payable]` 或 `fallback(bytes calldata input) external [payable] returns (bytes memory output)`
- 会在**函数发生错误调用时自动执行**（如函数名大小写拼写错误）
- 也可在合约收到以太币时自动执行，但需满足两个条件：
  1. 回退函数有 `payable` 修饰词
  2. 合约**不存在** `receive()` 函数
- 早期版本使用 `function() external [payable]` 声明，是合约中唯一一个没有函数名的函数

**(3) 接收函数（receive）**

- 一个智能合约**最多只能有一个**接收函数
- 声明方式：`receive() external payable`
- 会在**接收到以太币转账的时候自动执行**
- 早期 Solidity 版本中不存在 `receive()` 函数，因此回退函数是唯一一个能在收到以太币时自动执行的函数

**(4) 函数修改器（modifier）**

- 通常用于**限制函数的行为**
- 一个函数只有通过了函数修改器的检查才能被继续执行
- 典型用法：`onlyOwner` 修改器，限制函数只能由合约提交者运行

#### 4. 特殊变量与函数

**msg 相关（消息信息）：**

| 变量 | 含义 |
|------|------|
| `msg.value` | 合约消息所带的以太币，单位为 **wei** |
| `msg.sender` | 合约消息发送者（当前调用的直接发送方） |
| `msg.data` | 合约消息的 calldata |
| `msg.gas` | 合约消息剩余可用 gas |

**block 相关（区块信息）：**

| 变量 | 含义 |
|------|------|
| `block.blockhash(区块号)` | 近 256 个区块哈希 |
| `block.coinbase` | 合约交易所处区块的矿工地址 |
| `block.difficulty` | 合约交易所处区块的挖矿难度 |
| `block.gaslimit` | 合约交易所处区块的 gasLimit |
| `block.number` | 合约交易所处区块的高度 |
| `block.timestamp` | 合约交易所处区块的时间戳 |

**tx 相关（交易信息）：**

| 变量 | 含义 |
|------|------|
| `tx.gasprice` | 合约交易的 gasPrice |
| `tx.origin` | 合约交易的 from（完整调用链的最初发起者） |

**其他：**

| 变量/函数 | 含义 |
|------|------|
| `gasleft()` | 合约交易剩余可用 gas = gasLimit - 当前 gasUsed |
| `now` | 同 `block.timestamp` |
| `this` | 当前合约，可以明确转换为地址类型 |
| `selfdestruct(地址)` | 删除合约代码，并把余额返回到特定地址。调用后字节码被 `0x` 代替，但该地址仍可接收以太币，所有发送到该地址的以太币将永久丢失 |
| `keccak256()` | 哈希运算函数 |
| `abi.encodePacked()` | ABI 紧凑编码 |

**随机数生成：**
- 利用 `block.difficulty` 和 `block.timestamp` 结合 `msg.sender` 通过 `keccak256` 生成伪随机数
- **不安全**：矿工能够一定程度影响交易发送的时间（大概 900 秒），一定程序上控制随机数

**以太币发送方式对比（transfer / send / call）：**

| 函数 | 失败行为 | gas 限制 | 能否发到合约地址 |
|------|---------|---------|:---:|
| `transfer` | 回滚整个交易 | 2300 gas | 否（2300 gas 不足以支持基本操作） |
| `send` | 返回 false | 2300 gas | 否 |
| `call` | 返回 bool | 可自定义 | 能，但容易出现重入等安全问题 |

#### 5. 事件日志（Event Log）

**基本概念：**
- 以太坊中，合约的状态数据存储于状态树中，运行结果也写入状态
- 如果只依赖状态树记录，上层应用获取合约运行过程的某些状态时成本较高
- 事件日志用于记录合约运行过程中的关键动作，有别于状态树

**事件日志的特点：**

| 对比维度 | 状态树（State） | 事件日志（Event） |
|---------|-------------|---------------|
| 读写性 | 可读可写 | **单向输出、不可被合约读取** |
| Gas 定价 | 较高 | 更便宜 |
| 存储位置 | 状态树 | 收据树（Receipt Tree） |
| 写入方式 | 直接赋值 | `emit` 关键字 |

**事件声明与使用：**
```solidity
event ETHReceipt(address indexed sender, uint256 amount);

// 在函数中触发事件
emit ETHReceipt(msg.sender, msg.value);
```

**底层机制：**
- 事件及其参数类型被哈希编码为 **topic**（与函数签名类似，同样的事件名具有同样的 topic 值）
- 每个区块中，所有交易收据都将经过计算生成**布隆过滤器（Bloom Filter）**，记录为区块中 `logsBloom` 的值
- 布隆过滤器**不可以判断出某个值一定存在**某个集合中，但**可以判断出某个值一定不存在**某个集合中
- 上层应用只需向以太坊节点请求区块的 `logsBloom` 值，判断某个合约的事件是否一定没有发生，若不确定再请求详细的事件收据

> 核心结论：状态树与收据树本身并没有完整记录在区块中，它们由区块链的交易进行改变，而把**树根的值**、**布隆过滤器的值**记录到区块中。

#### 6. 错误处理

三种错误处理方式：

| 方式 | 语法 | 特点 |
|------|------|------|
| **require** | `require(条件参数, [消息参数])` | 条件为 false 则撤销状态更改，**返还剩余 gas**；用于判断某一条件是否满足（如输入验证） |
| **revert** | `revert([消息参数])` | 终止运行并撤销状态更改，**返还剩余 gas**；可配合自定义 error 使用 |
| **assert** | `assert(条件参数)` | 条件为 false 则使当前交易没有效果，**消耗交易剩余的 gas**；用于判断某种错误是否发生（如除 0 错误） |

**使用场景区分：**
- `require` 用于判断某一条件是否满足（如权限检查、输入验证）
- `revert` 用于需要传递复杂错误信息的场景
- `assert` 用于判断某种错误是否发生（如溢出、除 0 等内部错误）

**try...catch 机制：**
- 由于合约的**原子性**，一旦异常发生，交易将被回滚
- 在某些情况下，需要即使错误发生也不回滚交易，可以用 `try...catch`
- 用于捕获外部合约调用的异常

### 概述

- 基于区块链智能合约的 DApp 具有广阔的应用前景

### 6.3.1 私有链搭建

**步骤一：初始化创世区块**
```bash
geth --datadir mydata init genesis.json
```
- `--datadir`：指定区块链数据存储目录（mydata 文件夹）
- `genesis.json`：创世区块配置文件

**步骤二：启动以太坊节点**
```bash
geth --datadir mydata --networkid 666 --rpc --rpccorsdomain "*" --allow-insecure-unlock console
```
- `--networkid 666`：指定区块链网络 ID 为 666（避免与互联网中其他节点互相干扰）
- `--rpc`：打开以太坊节点的 RPC 接口，默认端口 **8545**
- `--rpccorsdomain`：允许浏览器在任意地址进行跨域调用
- `--allow-insecure-unlock`：允许通过 HTTP 解锁账户
- `console`：打开控制台，运行节点之后监听用户输入

**控制台说明：** 控制台是一个连接了节点的 **JavaScript 控制台**，自带 **web3.js** 库，可通过该库对以太坊节点的信息进行查询。

**步骤三：新建账号**
```javascript
personal.newAccount("")
```
- 括号内的参数为密码（置为空字符串方便实验）
- 私钥生成并保存于数据目录下
- 控制台输出新建账号的公钥地址

**步骤四：查看账户**
```javascript
eth.accounts      // 获取节点中保存的账号地址
eth.getBalance(eth.accounts[0])  // 查看余额
```

**步骤五：挖矿获取以太币**
```javascript
miner.start(); admin.sleep(1); miner.stop()
```
- 开启挖矿，等待 1 个区块的产生，结束挖矿
- 挖矿所用的 Coinbase 地址默认为本地账号中的第一个
- 首次挖矿时 geth 客户端将生成挖矿所需的 **DAG**，时间视机器配置而定
- 极少数情况下可能出现挖不到块的情况，请删除数据目录重新从 init 命令开始

### 6.3.2 智能合约部署与测试

**步骤：**

1. 在 Remix 中编写合约（如 Share.sol），编译通过
2. 在 "Deploy & run transactions" 中设置 Environment 为 **"External Http Provider"**，默认 8545 端口
3. 在节点控制台解锁账号：`personal.unlockAccount(eth.accounts[0], "", 3600)`（以空字符串解锁账号 3600 秒）
4. 在 Remix 中点击 Deploy，发送部署交易
5. 在节点控制台查看待打包的交易：`txpool.status`
6. 运行区块打包命令：`miner.start(); admin.sleep(1); miner.stop()`
7. 挖矿完成后，合约被部署到区块链上
8. 调用合约函数进行测试

**关键观察：**
- 交易发送后、区块打包前，调用合约的 getter 函数，合约状态并未改变
- 区块打包后，合约状态才改变

### 6.3.3 使用 Web3.js 调用智能合约

**Web3.js** 是以太坊官方支持开发的 **JavaScript 接口库**，可用于浏览器与以太坊节点的通信。

**开发流程：**

1. **引入 Web3.js 库**：`<script src="web3.min.js"></script>`

2. **新建 Web3 Provider**：根据 RPC 接口地址连接节点

3. **新建合约对象**：
   - 根据合约 **ABI** 及**地址**新建合约对象
   - 合约地址可从 Remix 控制台的交易信息中得到，或在节点控制台中查询交易收据得到

4. **调用合约函数**：
   - **读操作**：`Share.methods.getLength().call(callback)` — Web3.js 自动编码为 RPC 请求
   - **写操作**：`Share.methods.NewShare("数据").send({from: account})` — 发送交易

5. **事件监听**：
   - 通过开启 geth 的 WebSocket 端口并相应地在网页中监听该端口订阅合约事件
   - 可在合约更新时收到相应的事件通知，不必时时刻刻进行链上智能合约的查询，节省成本

**DApp 主流使用方式：**
- "**轻钱包 + 第三方节点**"方式
- 用户不需要运行自己的以太坊节点，只需保管好自己的钱包私钥
- 使用轻钱包中的浏览器（或浏览器轻钱包插件，如 **MetaMask**）使用 DApp
- 大大降低了区块链应用的使用门槛，但也带来了一定的**中心化问题**

## ⚖️ 重点对比 / 易混区分

### 1. 函数可见性对比

| 修饰符 | 合约内 | 子合约 | 合约外 |
|:---:|:---:|:---:|:---:|
| `public` | Y | Y | Y |
| `private` | Y | N | N |
| `internal` | Y | Y | N |
| `external` | N | N | Y（仅通过 `this.f()` 或外部调用） |

### 2. 函数行为修饰符对比

| 修饰符 | 读状态 | 写状态 | 接收以太币 |
|:---:|:---:|:---:|:---:|
| 无修饰 | Y | Y | N |
| `view` / `constant` | Y | N | N |
| `pure` | N | N | N |
| `payable` | Y | Y | Y |

### 3. 特殊函数对比

| 函数 | 数量限制 | 触发时机 | 声明方式 |
|------|:---:|------|------|
| **constructor** | 最多 1 个 | 合约部署时执行一次，字节码不上链 | `constructor() { ... }` |
| **fallback** | 最多 1 个 | 错误调用时 / 收到以太币时（当无 receive） | `fallback() external [payable]` |
| **receive** | 最多 1 个 | 收到以太币转账时 | `receive() external payable` |

### 4. 错误处理对比

| 方式 | 返还剩余 Gas | 适用场景 | 可附带消息 |
|:---:|:---:|------|:---:|
| `require` | 是 | 验证外部输入、权限检查 | 是 |
| `revert` | 是 | 复杂条件判断、自定义错误 | 是（支持自定义 error） |
| `assert` | **否（全部消耗）** | 内部错误检查（溢出、除 0） | 否 |

### 5. 以太币发送方式对比

| 方式 | Gas 限制 | 失败行为 | 安全性 | 能否发到合约 |
|:---:|:---:|------|------|:---:|
| `transfer` | 2300 | 回滚交易 | 较安全 | 否 |
| `send` | 2300 | 返回 false | 需检查返回值 | 否 |
| `call` | 可自定义 | 返回 bool | 注意重入攻击 | 能 |

### 6. msg.sender vs tx.origin

| 变量 | 含义 | 安全建议 |
|------|------|------|
| `msg.sender` | 当前调用的**直接**发送方 | 用于权限验证 |
| `tx.origin` | 完整调用链的**最初**发起者 | 不应用于权限验证（易受钓鱼攻击） |

### 7. 状态树 vs 事件日志

| 维度 | 状态树（Storage） | 事件日志（Event） |
|------|-------------|---------------|
| 存储位置 | 状态树（State Trie） | 收据树（Receipt Trie） |
| 可读性 | 合约可读取 | 合约不可读取（单向输出） |
| Gas 成本 | 高 | 低 |
| 写入方式 | 变量赋值 | `emit` 关键字 |
| 记录到区块 | 状态根（stateRoot） | logsBloom（布隆过滤器） |

### 8. storage vs memory

| 存储位置 | 说明 | 持久性 | Gas 成本 |
|------|------|:---:|:---:|
| `storage` | 存储在状态树中 | 持久化 | 高 |
| `memory` | 存储在内存中 | 函数执行完毕后释放 | 低 |

### 简答/综合题预测

1. **智能合约的三大特性是什么？中心化智能合约存在哪些弊端？区块链如何解决这些弊端？**
   - 三大特性：事件驱动、价值转移、自动执行
   - 弊端：容易被篡改、出错后难以追溯恢复（根源在于中心化执行环境）
   - 区块链解决方案：去中心化、多方验证、难以篡改

2. **简述 Solidity 中 constructor、fallback、receive 三种特殊函数的区别与适用场景。**
   - constructor：部署时执行一次，用于初始化状态（如设置 owner）
   - fallback：错误调用时或收到以太币时（无 receive 时）执行
   - receive：收到以太币转账时执行（比 fallback 更明确）

3. **对比 require、revert、assert 三种错误处理方式的区别和使用场景。**
   - require：返还 gas，用于外部输入验证
   - revert：返还 gas，用于复杂条件 + 自定义错误
   - assert：消耗全部剩余 gas，用于内部错误

4. **对比 transfer、send、call 三种以太币发送方式的区别。**
   - transfer/send：固定 2300 gas，无法发送到合约地址；transfer 失败回滚，send 返回 false
   - call：可自定义 gas，能发送到合约地址，但需注意重入攻击

5. **事件日志与状态树的区别是什么？布隆过滤器在其中的作用是什么？**
   - 状态树可读可写，Gas 高；事件日志单向输出，Gas 低
   - 布隆过滤器可快速判断某事件一定未在区块中发生，降低查询成本

6. **简述 DApp 开发的基本流程。**
   - 搭建区块链环境（私有链/测试网）
   - 使用 Solidity 编写智能合约（Remix）
   - 编译生成 ABI 和字节码
   - 部署合约到区块链
   - 通过 Web3.js 构建前端与合约交互
   - 可选：使用 MetaMask 等轻钱包降低使用门槛

7. **mapping 类型的底层存储特点是什么？与数组有何区别？**
   - mapping 存储键的 keccak256 哈希值而非键本身，减小存储体积
   - mapping 没有长度属性，数组有 length
